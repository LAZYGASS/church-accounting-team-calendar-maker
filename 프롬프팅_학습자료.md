# 프롬프팅 학습 자료: 예산집행 캘린더 프로젝트

## 개요
이 문서는 교회 재정집행 캘린더 PPT 자동화 프로젝트를 진행하면서 사용한 프롬프트와 AI의 응답 패턴을 분석한 학습 자료입니다.

---

## 1. 효과적인 프롬프팅 원칙

### 1.1 명확성 (Clarity)
**원칙**: 요구사항을 구체적이고 명확하게 표현

#### 예시 1: 불명확한 프롬프트 ❌
```
"폰트 바꿔줘"
```
**문제점**: 어떤 폰트로, 어디에 적용할지 불명확

#### 예시 2: 명확한 프롬프트 ✅
```
"파이썬 파일을 만들었는데 요청한거랑 다른게 있어
모든 폰트는 나눔 스퀘어 볼드야"
```
**장점**:
- 대상 명확 (모든 폰트)
- 변경 내용 구체적 (나눔 스퀘어 볼드)
- 현재 상태 언급 (이미 파일이 있음)

---

### 1.2 단계적 요구사항 (Incremental Requirements)
**원칙**: 한 번에 하나씩, 복잡한 요구사항은 여러 단계로 나누기

#### 대화 흐름 예시:

**1단계: 기본 기능**
```
프롬프트: "모든 폰트는 나눔 스퀘어 볼드야"
→ 응답: 폰트 변경 완료
```

**2단계: 요일 및 색상**
```
프롬프트: "요일은 영어 줄임이고 글씨는 일요일은 지금처럼 핑크
평일은 검정 토요일은 하늘색이야"
→ 응답: 요일 표시 및 색상 변경
```

**3단계: 테두리**
```
프롬프트: "날짜들사이에는 파란색으로 칸이 나눠져있어야하고"
→ 응답: 테두리 추가
```

**4단계: 비즈니스 로직**
```
프롬프트: "예산집행일은 둘째 넷째 화요일이고
예산집행일이라는 도형은 항상 그 날짜 밑에 위치해야해"
→ 응답: 로직 변경 및 도형 위치 조정
```

**장점**:
- AI가 각 단계를 정확히 처리
- 오류 발생 시 쉽게 롤백
- 진행 상황 파악 용이

---

### 1.3 맥락 제공 (Context Provision)
**원칙**: 현재 상태와 문제점을 함께 설명

#### 예시 1: 맥락 없음 ❌
```
"도형 위치 바꿔"
```

#### 예시 2: 맥락 포함 ✅

**포함된 맥락**:

### 1.5 복합 요구사항의 구조화
**원칙**: 여러 요구사항을 논리적으로 구조화하여 전달

#### 예시: 복합 프롬프트
```
"운영위원회 폰트 14 밑줄 황금색보기 지난달 날짜도 표시 #C0D3DE
예산지급일 표시 도형및 글짜 하나의 도형으로 묶음
```

   - 문제점 명시 (현재 너무 내려가있음)

**AI 처리 방식**:
- 각 요구사항을 개별 작업으로 분리
- 우선순위에 따라 순차 처리
- 관련된 코드 블록을 한 번에 수정

## 2. 프롬프트 패턴 분석

### 2.1 문제 보고 패턴

#### 패턴: "X가 안되어 있어"
```
사용자: "왜 표에 표선이 없지 파란색으로 있어야해"
```

**AI 해석**:
- 문제 인식: 표에 선이 없음
- 해결 방향: 파란색 테두리 추가
- 원인 분석: 테두리 함수 확인

**대응 전략**:
1. 현재 코드 확인 (Read)
2. 문제 원인 파악
3. 해결책 구현 (Edit)
4. 결과 설명

---

### 2.2 디자인 요구사항 패턴

#### 패턴: "A는 B처럼, C는 D로"
```
사용자: "요일은 영어 줄임이고 글씨는
일요일은 지금처럼 핑크 평일은 검정 토요일은 하늘색이야"
```

**구조화**:
| 요소 | 조건 | 결과 |
|------|------|------|
| 요일 형식 | - | 영어 줄임 (Sun, Mon...) |
| 일요일 | 요일 = 일요일 | 핑크 |
| 평일 | 요일 = 월~금 | 검정 |
| 토요일 | 요일 = 토요일 | 하늘색 |

**AI 구현**:
```python
if col_idx == 0:  # 일요일
    run.font.color.rgb = COLOR_SUNDAY
elif col_idx == 6:  # 토요일
    run.font.color.rgb = COLOR_SATURDAY
else:  # 평일
    run.font.color.rgb = COLOR_BLACK
```

### 2.3 비즈니스 로직 변경 패턴

#### 패턴: "기존 X를 Y로 변경"
```
사용자: "예산집행일은 둘째 넷째 화요일이고"
```

**변경 사항**:
- 기존: 둘째/넷째 월요일
- 신규: 둘째/넷째 화요일

**AI 처리**:
1. 요일 인덱스 변경 (1 → 2)
2. 변수명 변경 (monday_weeks → tuesday_weeks)
3. 주석 업데이트
4. 관련 계산 로직 조정

### 2.4 위치 조정 패턴

#### 패턴: "X가 Y에 위치해야 해, 지금은 Z 문제가 있음"
```
사용자: "예산집행일이라는 도형은 항상 그 날짜 밑에 위치해야해
지금은 너무 내려가있음"
```

**분석**:
- 목표 위치: 날짜 셀 바로 밑
- 현재 문제: 너무 내려가 있음
- 요구: 간격 최소화

**AI 해결**:
```python
# 기존
box_top = cell_bottom + Inches(0.05)

# 수정
box_top = cell_bottom - Inches(0.02)  # 셀 바로 아래
```

## 3. 고급 프롬프팅 기법

### 3.1 암시적 요구사항 파악하기

#### 사례 1: "요청한거랑 다른게 있어"
```
프롬프트: "파이썬 파일을 만들었는데 요청한거랑 다른게 있어
모든 폰트는 나눔 스퀘어 볼드야"
```

**암시된 내용**:
- ✅ 파일이 이미 존재함
- ✅ 이전에 요청했던 내용이 있음
- ✅ 폰트만 다름 (다른 것은 OK)
- ✅ 파일 전체를 다시 만들 필요 없음 (Edit만 필요)

**AI의 올바른 대응**:
1. 파일 읽기 (Read)
2. 폰트 부분만 수정 (Edit)
3. 다른 부분은 유지

---

### 3.2 복합 조건 표현

#### 사례: 날짜 색상 우선순위
```
프롬프트: "날짜들사이에는 파란색으로 칸이 나눠져있어야하고,
예산집행일은 둘째 넷째 화요일이고"
```

**조건 계층**:
```
1순위: 결재일 → 핑크 볼드
2순위: 집행일 → 파란 볼드
3순위: 요일별 기본 색상
   - 일요일 → 핑크
   - 토요일 → 하늘색
   - 평일 → 검정
4순위: 이전 달 → #C0D3DE
```

**AI 구현**:
```python
if is_prev_month:
    color = PREV_MONTH
elif day in approval_days:
    color = PINK
elif day in execution_days:
    color = BLUE
else:
    if col_idx == 0:
        color = SUNDAY
    elif col_idx == 6:
        color = SATURDAY
    else:
        color = BLACK
```

**특징**:

- 맥락 의존적 (지금까지 = 전체 대화)
- 암시적 요구사항 (변경 내역 정리)

**AI의 해석**:
- "지금까지" = 전체 대화 기록
- "내역" = 변경 사항 요약
- "md파일" = 마크다운 형식

**출력물**:
- 시간순 변경 내역
- 기술적 세부사항
- 색상 팔레트 표
- 사용 방법

---

### 6.2 PRD 요청

**프롬프트**:
```
"prd만들어줘"
```

**특징**:
- 극도로 간결 (3단어)
- 전문 용어 사용 (PRD)
- 맥락 완전 의존

**AI의 이해**:
- PRD = Product Requirements Document
- 프로젝트 전체 범위
- 표준 PRD 구조 적용

**생성된 섹션**:
1. Executive Summary
2. 사용자 정의
3. 문제 정의
4. 제품 요구사항
5. 기술 사양
6. UI/UX 요구사항
7. 성공 지표
8. 제약사항
9. 출시 계획

**핵심 학습**:
- 전문 용어 사용 시 표준 템플릿 적용
- 맥락이 충분하면 짧은 프롬프트도 효과적
- AI가 암시적 요구사항 추론

### 6.3 학습 자료 요청

**프롬프트**:
```
"나중에 프롬프팅 공부할수있게도 md"
```

**특징**:
- 메타적 요청 (프롬프트에 대한 프롬프트)
- 교육적 목적
- 재귀적 성격

**AI의 해석**:
- 대화 전체를 분석 대상으로
- 프롬프팅 기법 추출
- 패턴 및 원칙 정리
- 사례 연구 포함

**출력물** (현재 문서):
- 프롬프팅 원칙
- 패턴 분석
- 좋은/나쁜 예시
- 실전 팁

## 7. 프롬프팅 체크리스트

### 작성 전 확인사항
```
□ 무엇을 원하는지 명확한가?
□ 현재 상태를 언급했는가?
□ 구체적인 값을 제시했는가?
□ 문제점을 설명했는가?
□ 우선순위가 있는가?
```

### 복잡한 요청 시
```
□ 여러 요구사항을 나열했는가?
□ 각 요구사항이 명확한가?
□ 서로 모순되지 않는가?
□ 예시를 제공할 수 있는가?
```

### 문제 해결 시
```
□ 무엇이 문제인지 설명했는가?
□ 기대하는 결과를 명시했는가?
□ 시도해본 방법을 언급했는가?
```

## 8. 레벨별 프롬프팅 가이드

### 🌱 초급 (Beginner)
**목표**: 명확한 한 가지 요청하기

**예시**:
```
"폰트를 나눔 스퀘어 볼드로 변경해줘"
```

**포인트**:
- 한 번에 한 가지만
- 구체적인 값 제시
- 대상 명확히

---

### 🌿 중급 (Intermediate)
**목표**: 맥락과 함께 복합 요청하기

**예시**:
```
"요일은 영어 줄임이고 글씨는
일요일은 핑크 평일은 검정 토요일은 하늘색이야"
```

**포인트**:
- 여러 조건 나열
- 현재 상태 언급
- 논리적 구조

---

### 🌳 고급 (Advanced)
**목표**: 암시적 요구사항 포함한 복합 요청

**예시**:
```
"예산집행일이라는 도형은 항상 그 날짜 밑에 위치해야해
지금은 너무 내려가있음"
```

**포인트**:
- 문제점 진단
- 원하는 결과 명시
- 현재 상태와 비교
- 암시적 제약사항 포함

---

### 🏆 전문가 (Expert)
**목표**: 메타적 요청 및 구조화된 결과물

**예시**:
```
"prd만들어줘"
"나중에 프롬프팅 공부할수있게도 md"
```

**포인트**:
- 전문 용어 활용
- 맥락 완전 의존
- 표준 산출물 기대
- 암시적 구조 이해

## 9. 실전 연습 문제

### 연습 1: 프롬프트 개선하기

**원본 프롬프트** ❌
```
"색 바꿔줘"
```

**개선된 프롬프트** ✅
```
"일요일 날짜 색상을 핑크(#FD79A8)로 변경해줘"
```

**개선 포인트**:
- 대상 명시 (일요일 날짜)
- 구체적 색상 코드 (#FD79A8)
- 작업 명확 (색상 변경)

---

### 연습 2: 복합 요청 구조화하기

**원본 프롬프트** ❌
```
"표도 고치고 폰트도 바꾸고 색도 바꿔줘"
```

**개선된 프롬프트** ✅
```
"표 테두리를 파란색(#0061FF) 1.5pt로 변경하고,
모든 폰트를 나눔스퀘어 볼드로 바꾸고,
일요일은 핑크, 토요일은 하늘색으로 표시해줘"
```

**개선 포인트**:
- 각 요청 구체화
- 값 명시
- 우선순위 정리

---

### 연습 3: 문제 진단 요청

**원본 프롬프트** ❌
```
"안돼"
```

**개선된 프롬프트** ✅
```
"왜 표에 테두리가 표시되지 않지?
파란색 선으로 칸이 구분되어야 해"
```

**개선 포인트**:
- 문제 명확히 (테두리 없음)
- 기대 결과 (파란 선)
- 진단 요청 (왜)

## 10. 핵심 요약

### 효과적인 프롬프트의 5가지 요소

1. **명확성 (Clarity)**
   - 무엇을, 어떻게, 왜

2. **구체성 (Specificity)**
   - 숫자, 색상 코드, 정확한 값

3. **맥락 (Context)**
   - 현재 상태, 문제점, 목표

4. **구조 (Structure)**
   - 논리적 순서, 우선순위

5. **검증 가능성 (Verifiability)**
   - 결과 확인 가능한 기준

---

### 프롬프팅 성장 경로

```
1단계: 한 가지 명확한 요청
    ↓
2단계: 맥락 포함한 요청
    ↓
3단계: 복합 조건 표현
    ↓
4단계: 암시적 요구사항 활용
    ↓
5단계: 메타적 요청 및 구조화
```

---

### 마지막 조언

> "완벽한 프롬프트는 없다.
> 반복적 개선과 명확한 피드백이 핵심이다."

**기억할 점**:
- AI는 당신의 의도를 추론하지만, 명확할수록 좋다
- 짧은 프롬프트도 좋지만, 맥락이 중요하다
- 문제가 있으면 즉시 피드백하라
- 단계적으로 개선하는 것이 효과적이다
